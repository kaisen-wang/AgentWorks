# AI Agent 工作集 – 需求开发文档

## 1. 项目背景与目标

### 1.1 项目名称
**AgentWorks** – 一人公司 AI Agent 工作集工具

### 1.2 项目愿景
为一人公司/独立创业者提供**类聊天软件**的 AI Agent 管理平台。用户可通过自然语言与单个 Agent 对话、拉群协作，并基于**企业级组织架构**（有限管理幅度、双向工作流、标准化行为）来放大个人管理半径，实现“一个人 + AI 组织”的高效运作。

### 1.3 目标用户
- 一人公司创始人 / 独立开发者 / 自由职业者
- 需要同时管理市场营销、产品设计、客户服务、行政财务等多个职能
- 技术背景一般，希望用聊天式交互完成复杂任务
- 对成本敏感，需要可控的 AI 调用预算

---

## 2. 用户角色

| 角色 | 说明 | 典型权限 |
|------|------|----------|
| **老板（唯一用户）** | 一人公司的决策者，系统的唯一人类用户 | 所有权限：创建/删除 Agent，设定组织架构，跨级干预，查看所有归档，配置系统 |
| **外部协作者** | 临时邀请的外包、客户或合作方（真人或外部 AI） | 仅限被加入的群聊，不可查看组织架构、全局归档、审计日志 |
| **Agent（内部）** | 用户创建的虚拟员工 | 根据组织架构被授予执行、汇总、上报、归档权限；不能越级访问 |

## 3. 核心功能需求

### 3.1 组织架构管理

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| ORG-01 | 用户可通过自然语言创建 Agent（如“创建一名营销主管 Agent，使用 deepseek-v4-flash”），系统自动分配唯一标识和头像 | P0 |
| ORG-02 | 支持设定 Agent 之间的上下级关系，形成树形组织架构。一个 Agent 只能有一个直接上级 | P0 |
| ORG-03 | **管理幅度限制**：每个 Agent 的直接下属数量上限可配置（默认 5）。超过上限时系统阻止添加并提示 | P0 |
| ORG-04 | 用户可随时调整组织架构（变更上级、删除 Agent、修改上限），调整后历史任务不受影响 | P1 |
| ORG-05 | 提供可视化组织架构图（侧边栏或弹窗），展示 Agent 层级关系及管理幅度使用情况 | P1 |
| ORG-06 | 支持“一键克隆”整个组织架构（包括 Agent 角色、上下级关系、剧本、知识库），用于多项目或多客户隔离 | P2 |

### 3.2 Agent 标准化行为（四动作）

**定义**：每个 Agent（无论主管或专员）都必须实现以下四个基础动作，形成闭环。

| 动作 | 定义 | 输入 | 输出 |
|------|------|------|------|
| **执行** | 完成一个具体任务（生成内容、调用 API、查询数据等） | 任务描述（JSON 或自然语言） | 原始执行结果 |
| **汇总** | 收集并合并多个结果（如下属上报、自身多次执行） | 结果列表 | 结构化摘要（文本/表格/卡片） |
| **上报** | 将结果、问题或决策请求发送给上级（或用户） | 上报内容 + 目标（默认为直接上级） | 无阻塞发送 |
| **归档** | 将任务输入、中间状态、输出、元数据保存到持久存储 | 归档记录（含时间戳、来源、内容） | 归档 ID |

**需求**：
- ACT-01: 每个 Agent 必须实现这四个方法的接口，但允许某些动作为空操作（如“汇总”直接返回第一个结果） | P0
- ACT-02: 系统提供基类 `BaseAgent`，开发者（或通过自然语言配置）可继承并实现定制逻辑 | P0
- ACT-03: 用户可在聊天界面中查看任一 Agent 的当前动作状态（执行中/汇总中/已上报/已归档） | P1
- ACT-04: 支持为每个动作设置超时时间（默认 30 秒），超时后触发异常上报 | P1

### 3.3 双向工作流

#### 3.3.1 从上到下（Top-down）

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| TDN-01 | 用户可在群聊或单聊中向一个主管 Agent 下达宏观任务（自然语言）。系统将该消息标记为“任务” | P0 |
| TDN-02 | 主管 Agent 能自动拆解任务为子任务，并分配给下属 Agent（基于下属的能力标签） | P0 |
| TDN-03 | 任务拆解过程对用户可见（可选，用户可点击任务卡片查看树形拆解图） | P1 |
| TDN-04 | 用户可在任何层级直接修改或补充任务（如“把风格改成复古”），修改后的指令向下传递覆盖原子任务 | P1 |
| TDN-05 | 支持用户将完整任务流程（包括拆解规则、分配关系、上报规则）保存为“剧本”，后续可通过自然语言实例化执行 | P1 |

#### 3.3.2 从下到上（Bottom-up）

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| BUP-01 | 下属 Agent 在执行中遇到无法处理的问题（权限不足、需要决策、资源缺失）时，自动上报给直接上级 | P0 |
| BUP-02 | 上报内容结构化：包含“问题描述 + 已尝试方案 + 建议决策选项”。上级（或用户）可通过点击按钮快速决策 | P0 |
| BUP-03 | 支持进度/结果的自发汇总上报（如里程碑完成），用户可配置上报频率（每日/每周/完成时） | P1 |
| BUP-04 | 上报链可配置：用户可为每个主管设定“决策阈值”（如预算 < 5 元自动批准，无需上报），减少打断 | P1 |
| BUP-05 | 当用户（老板）长时间未响应上报时，支持自动升级策略（如1小时后按默认选项执行，或再次提醒） | P2 |

### 3.4 聊天界面（核心交互）

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| UI-01 | 主界面类似即时通讯软件：左侧组织/会话列表，右侧聊天窗口 | P0 |
| UI-02 | 支持单聊（与单个 Agent）和群聊（多个 Agent + 用户）。群聊可跨组织架构临时创建 | P0 |
| UI-03 | 消息类型支持：纯文本、**任务卡片**（含负责人/截止时间/状态/子任务列表）、**上报卡片**（带决策按钮）、进度条、文件/图片 | P0 |
| UI-04 | 支持回复某条消息（形成线程），Agent 需能感知引用关系 | P1 |
| UI-05 | 支持斜杠命令：`/new_task`, `/summary`, `/archive`, `/help` 等，Agent 解析并执行 | P1 |
| UI-06 | 所有消息支持全文搜索（包括归档内容） | P2 |

### 3.5 数据与知识管理

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| KNL-01 | **分层知识库**：全局（所有 Agent 只读）、部门/角色（特定 Agent 可读写）、个人（仅该 Agent） | P1 |
| KNL-02 | 用户可通过自然语言更新知识库内容（如“告诉营销主管我们的品牌色是#00FF00”） | P1 |
| KNL-03 | **归档检索**：支持自然语言查询历史任务（如“上个月那个视频任务为什么延迟？”），Agent 返回带引用来源的答案 | P1 |
| KNL-04 | 归档支持按时间、Agent、任务类型、异常标签过滤，并支持导出为 JSON/CSV | P2 |
| KNL-05 | 提供成本统计模块：每个 Agent、每个任务的 API 调用次数及费用（支持 OpenAI 及其他常见模型） | P1 |

### 3.6 可靠性与容错

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| RFT-01 | 每个任务可设置超时（继承自上级或默认 30s）。超时后上级 Agent 自动重试（最多 3 次，指数退避）或上报 | P0 |
| RFT-02 | 支持 dry-run 模式：敏感操作（如发布、发送邮件）先模拟执行，用户确认后才真正执行 | P1 |
| RFT-03 | 系统定期 ping 每个 Agent（心跳）。连续失败后，上级 Agent 尝试重启或上报用户，并提供替代建议 | P2 |
| RFT-04 | 所有四个动作及上报链生成不可篡改的审计日志（时间戳、内容哈希），支持导出用于合规 | P2 |

### 3.7 扩展性与集成

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| EXT-01 | Agent 可声明所需外部工具/API（如发送邮件、调用 Midjourney）。用户通过聊天界面授权（一次性或按次） | P1 |
| EXT-02 | 内置插件市场，提供常用服务连接器（Gmail, Notion, Stripe, Twitter 等） | P2 |
| EXT-03 | 支持外部事件触发（Webhook、邮件、日历事件）启动 Agent 工作流。用户用自然语言配置触发器 | P2 |
| EXT-04 | 允许为同一角色部署多个 Agent 版本，支持 A/B 测试（流量比例分配、指标收集） | P3 |

### 3.8 一人公司特有功能

| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| SOLO-01 | **老板休息模式**：用户开启后，所有上报自动转给“值班主管 Agent”（按预设规则处理），用户恢复后查看报告 | P2 |
| SOLO-02 | **自然语言配置一切**：支持通过自然语言修改 Agent 配置（决策阈值、归档策略、知识库内容），无需后台界面 | P1 |
| SOLO-03 | **成本控制**：用户可为每个 Agent 设置月度预算（美元/tokens），超额时自动上报或降级到低成本模型 | P1 |
| SOLO-04 | **快速模板克隆**：一键复制整个工作区（组织架构+剧本+知识库），用于新项目启动 | P1 |
| SOLO-05 | 支持临时邀请外部协作者（真人或外公司 Agent），仅限群聊可见，无法访问组织架构和全局归档 | P1 |

## 4. 非功能性需求

| 类别 | 需求 | 指标 |
|------|------|------|
| 性能 | 单个 Agent 动作响应时间（不含外部 API） | ≤ 500ms |
| 性能 | 群聊消息（含任务卡片渲染）端到端延迟 | ≤ 2s |
| 可扩展性 | 支持同时运行 Agent 数量 | MVP: ≤ 50；后续目标: 200 |
| 可靠性 | 系统无单点故障（Agent 进程崩溃不影响其他 Agent） | Agent 进程隔离 |
| 安全性 | API 密钥存储加密（AES-256），传输 TLS 1.3 | 必需 |
| 可用性 | 用户无需编程知识即可创建和配置 Agent | 自然语言交互成功率 ≥ 90% |
| 可维护性 | 提供 Agent 行为日志（DEBUG 级别）和审计日志（INFO 级别） | 日志保留 30 天 |

## 5. MVP 范围与优先级

**P0 – 必须实现（核心可用）**  
- 创建 Agent（ORG-01）  
- 设定上下级 & 管理幅度上限（ORG-02, ORG-03）  
- Agent 四动作接口定义（ACT-01, ACT-02）  
- 从上到下：用户下达任务、主管拆解分配（TDN-01, TDN-02）  
- 从下到上：异常上报 + 决策卡片（BUP-01, BUP-02）  
- 聊天界面基础版（UI-01, UI-02, UI-03 文本+任务卡片）  
- 超时重试（RFT-01）  
- 自然语言修改简单配置（SOLO-02 部分）  

**P1 – 高价值，建议包含在 Alpha 阶段**  
- 可视化组织架构图（ORG-05）  
- 任务拆解树查看（TDN-03）  
- 剧本保存与实例化（TDN-05）  
- 知识库分层（KNL-01）  
- 归档自然语言检索（KNL-03）  
- 成本统计（KNL-05）  
- 外部协作者支持（SOLO-05）  
- 全局知识库更新（KNL-02）  

**P2 – 后续迭代**  
- 一键克隆组织架构（ORG-06）  
- 外部事件触发（EXT-03）  
- 老板休息模式（SOLO-01）  
- 插件市场（EXT-02）  
- 审计日志导出（RFT-04）

## 6. 附录：用户故事摘要（高保真场景）

> **张三** 一人公司老板，需要发布新产品壁纸“赛博山景”。  
> 1. 他用自然语言创建了 `营销主管`、`图文本设计`、`平台发布` 三个 Agent，并设定了上下级关系（营销主管管另外两个）。  
> 2. 在“营销作战室”群聊中，他 @营销主管 下达任务：下周五前在小红书、抖音、公众号发预热图文，风格霓虹+山水，限时 7 折。  
> 3. 营销主管自动拆解：分配设计任务给图文本设计，分配发布任务给平台发布，并回复张三已开始。  
> 4. 设计完成上报后，主管发现抖音文案超限，通过上报卡片让张三决策（选长图文模式）。  
> 5. 发布完成后，主管汇总链接上报给张三。整个过程张三只做了两次决策，其余自动。  
> 6. 一周后，张三通过 `/archive 查询“赛博山景”发布任务，显示费用汇总` 获得总成本 $1.23，并将本次流程保存为“新品宣发_标准流程”剧本。  
> 7. 他用自然语言设置决策阈值：“以后营销主管对于 5 元以内的修改自行批准”。  
> 8. 后续新项目只需说“运行剧本：新品宣发_标准流程，替换产品名为‘赛博山水’”。
