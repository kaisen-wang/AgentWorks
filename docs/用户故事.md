# 用户故事：一人公司老板「张三」用 AI Agent 工作集完成新产品发布

## 背景

- **用户**：张三，一人公司“智绘科技”创始人，做 AI 生成壁纸的生意。
- **目标**：低成本、高效率地完成一次新品（“赛博山景”系列壁纸）在小红书、抖音、公众号的同步宣发。
- **工具**：AI Agent 工作集 – 聊天界面 + 组织架构 + 双向工作流。

## 前置准备：搭建虚拟团队

张三打开工具，看到一个空白的聊天列表。他首先创建几个 Agent。

### 1. 创建 Agent（自然语言）

他在主聊天框输入：

> `/new_agent 图文本设计`  
> “这个 Agent 擅长根据主题生成海报、文案、排版。使用 DALL‑E 和 GPT‑4。”

系统自动生成一个 Agent，头像是一个画笔图标。

同样方式，创建 `平台发布` Agent（负责各平台格式适配与发布）、`营销主管` Agent（用来管理前两个）。

### 2. 设定组织架构

张三对系统说：

> “把 `营销主管` 设为 `图文本设计` 和 `平台发布` 的上级，管理上限 5。并且所有上报先给主管，主管无法解决再给我。”

系统在后台建立了树形结构：**张三（老板）→ 营销主管 → 图文本设计 / 平台发布**。

张三可以随时在聊天侧边栏看到这个组织架构图。

后来张三想再创建一个 `视频剪辑 Agent` 挂在营销主管下，负责短视频制作。他输入：

> `/new_agent 视频剪辑`，上级设为 `营销主管`。

系统提示："营销主管当前已有 2 个下属，管理幅度上限为 5，尚可添加。"创建成功。

张三继续尝试添加第 6 个下属时，系统阻止并提示："营销主管的管理幅度已达上限 5，请先调整架构或提升上限。"张三这才意识到管理幅度的存在——它防止一个主管被过多下属淹没，保证工作流质量。

### 3. 创建部门群聊

张三创建一个群聊：“**营销作战室**”，把 `营销主管` 拉进来。`营销主管` 自动将它的两个下属拉进群（但下属默认只读，除非被@）。

## 场景启动：从上到下分派任务

### 张三下达宏观指令

在“营销作战室”群中，张三输入：

> `@营销主管` 我们新产品“赛博山景”壁纸下周五上线。需要在小红书、抖音、公众号各发一篇预热图文。风格是“霓虹+山水”，强调限时 7 折。你来搞定。

### 主管 Agent 的自动拆解（执行+分配）

`营销主管` 收到后，内部执行（不直接在群里说）：
1. 识别出两个子任务：**创作内容**、**多平台发布**。
2. 发送私密消息给 `图文本设计`：
   > “请设计一套小红书、抖音、公众号配图及文案。风格要求：霓虹+山水，突出限时 7 折。下周三前交付草稿。”
3. 发送给 `平台发布`：
   > “等设计稿确认后，将内容适配各平台尺寸和字数限制，并在下周五前发布。发布后截图回传。”
4. 在群聊中回复张三（**上报上级**）：
   > “已分解任务：设计 Agent 负责图文，发布 Agent 负责上线。我会在下周三汇总草稿给您确认。”

张三收到确认消息，看到任务卡片（显示两个子任务进度 0%）。

## 从下到上的正常汇总与异常上报

### 正常汇总：设计 Agent 完成草稿

`图文本设计` 完成三套设计后，**执行**生成图片和文案。在生成过程中，DALL‑E 调用累计费用接近月度预算 $10 的 90%。系统自动上报张三：

> **【预算告警】**  
> 图文本设计本月 DALL‑E 预算已用 90%（$9.0 / $10.0）。  
> 建议：A) 增加额度至 $15；B) 后续调用切换至低精度模型（DALL‑E Mini）；C) 暂不处理，超额后自动降级。  
> 请选择： [A] [B] [C]

张三选择 **[C]**，让系统在超额后自动降级到低成本模型，不打断当前任务。设计 Agent 继续完成剩余生成。

设计 Agent **汇总** 为一个 ZIP 文件，附上预览链接，然后 **上报** 给直接上级 `营销主管`（同时 **归档** 到设计库）。

`营销主管` 收到后，**执行**检查：发现抖音文案字数超限（200 字 vs 限制 150 字）。它尝试让设计 Agent 自动缩短，但设计 Agent 反馈缩短后会丢失核心卖点。

于是 `营销主管` **上报** 给张三（在群聊中发送**上报卡片**）：

> **【需要决策】**  
> 问题：抖音文案超出限制 50 字。  
> 建议：A) 缩短文案，可能弱化“限时 7 折”；B) 改用抖音长图文模式（需额外 1 天制作）。  
> 请点击选择： [A] [B]  

张三正在喝咖啡，手机弹出一个带按钮的通知。他点了 **[B]**。

`营销主管` 收到决策，转告 `图文本设计` 执行长图文版本。最终设计稿满足要求。

### 执行发布

`平台发布` Agent 拿到最终设计稿，**执行** 分别发布到三个平台。

发布过程中，小红书 API 突然限流，`平台发布` 心跳超时未响应。`营销主管` 收到心跳告警，自动重试 3 次（指数退避：1s → 2s → 4s），均因限流失败。`营销主管` 随即上报张三：

> **【Agent 故障】**  
> 平台发布因小红书 API 限流未响应，重试 3 次均失败。  
> 替代方案：A) 仅先发布抖音+公众号，小红书稍后补发；B) 等待 30 分钟后重试；C) 暂停，稍后手动重试。  
> 请选择： [A] [B] [C]

张三选择 **[A]**，先完成其他平台。`平台发布` 跳过小红书，成功发布抖音和公众号。30 分钟后限流解除，`平台发布` 自动补发小红书。

发布完成后，`平台发布` **汇总** 各平台链接，**上报** 给 `营销主管`，并 **归档** 发布记录（含小红书补发记录）。

`营销主管` 再在群里 **汇总** 给张三：

> “✅ 已发布：  
> 小红书：链接1  
> 抖音：链接2（长图文）  
> 公众号：链接3  
> 所有内容已归档。需要我在一周后统计数据吗？”

张三回复：“好，一周后自动跑数据。”

## 跨层级干预（老板特权）

张三在群里看到抖音预览效果不太理想，他想临时改一张封面图。他不用通过主管，直接 @ `图文本设计`：

> `@图文本设计` 把抖音封面的霓虹灯亮度调高 20%，重新发布一下。

系统弹出提示：“这绕过了营销主管，是否通知主管？” 张三选“通知”。  
`图文本设计` 执行修改，重新输出。`平台发布` 收到更新后替换抖音封面。所有动作被记录到审计日志。

## 外部协作者（临时邀请）

张三觉得抖音长图文的排版还需要专业设计师把关，于是临时邀请外包设计师李华进群：

> `/invite @李华（外部）` 到"营销作战室"

系统提示："李华为外部协作者，仅可查看当前群聊，无法访问组织架构、全局归档及其他群聊。"张三确认。

李华进入群聊后，只能看到"营销作战室"中的消息和文件。她审阅了抖音长图文草稿，在群里回复修改建议。`图文本设计` 根据建议调整后，张三将李华移出群聊。李华的访问权限随即失效，历史消息对她不再可见。

## 归档与检索

一周后，张三想回顾上次发布花了多少成本。他在聊天框输入：

> `/archive 查询“赛博山景”发布任务，显示费用汇总`

系统从归档中提取数据：
- 设计 Agent：DALL‑E 调用 15 次，费用 $1.2
- 发布 Agent：API 调用 3 次，费用 $0.03
- 主管 Agent：无直接外部成本
- 总成本 $1.23，总耗时 3 天（从下达指令到发布完成）

张三满意，把这次完整流程保存为**剧本**，命名为“新品宣发_标准流程”。以后只需说“运行剧本：新品宣发_标准流程”，并替换产品名和日期即可。

## 自然语言配置与调整

张三觉得每次上报决策都要手动点按钮有点烦，尤其预算小的时候。他直接对系统说：

> “以后营销主管对于 5 元以内的修改决策不需要上报我，可以自行批准。”

系统更新 `营销主管` 的决策阈值。第二天另一个小任务出现时，主管自己做了决定，张三没有被打扰。

## 休息模式（可选）

周五晚上张三想完全休息。他开启“老板休息模式”，并设定值班主管 Agent（另一个通用 Agent）按照以下规则处理：
- 预算 < 10 元 → 自动执行
- 紧急事务 → 发送短信摘要，但不用等回复
- 其他事务 → 记录，周一早上列出清单

周一早上，张三查看值班报告，对 3 条自动执行的任务没有异议，继续工作。

## 结论

张三通过这个工具，用聊天的方式管理了一个**虚拟的营销部门**。组织架构和双向工作流让他既保留了掌控感，又不需要亲自做每个细节。一人公司真正实现了 **“一个人 + AI 组织”** 的运作模式。

> 整个用户故事覆盖了：创建 Agent、设定上下级、管理幅度（含超限拦截）、四个动作、群聊交互、从上到下拆解、从下到上异常上报/决策、成本控制（预算告警与自动降级）、心跳与容错（限流重试与替代方案）、外部协作者（临时邀请与权限隔离）、老板跨级干预、归档检索、剧本保存、自然语言配置、休息模式。所有之前讨论的需求都在故事中自然体现。
